## 1. システム概要
プロジェクト単位による作業を一元管理する

### 1-1. システム名

**TODO Manager**（作業リスト管理システム）

### 1-2. システム目的

複数のシステム・プロジェクトにまたがるチケット（タスク）を一元管理するシステム。
各チケットの進捗状態およびデプロイ状態を追跡し、開発チームの作業効率向上を支援する。

### 1-3. 対象ユーザー

| ロール     | 説明                                              |
|------------|---------------------------------------------------|
| 管理者 (ADMIN) | マスタ管理画面・全機能へのアクセス可              |
| 一般ユーザー (USER) | タスクリスト画面のみ利用可能（マスタ編集不可） |

### 1-4. システム全体像

```
[フロントエンド]            [バックエンド]            [データベース]
React 19 + TypeScript  <--> Spring Boot 3.5  <--> MySQL 8
(localhost:5173)            (localhost:8080)       (192.168.0.3:3306/tododb)
                                  |
                          [外部AIサービス]
                           Ollama (llama3)
                          (localhost:11434)
```

## 2. 業務要件

### 2-1. 解決する業務課題

- 複数プロジェクトのチケットをExcelや紙で管理していたため、状態把握が困難
- 進捗状態・デプロイ状態の更新が属人化していた
- マスタデータ（システム・プロジェクト・コード）の一元管理が不十分だった

### 2-2. 業務フロー

```
1. ログイン
   └─→ ユーザーIDとパスワードで認証
       └─→ 認証成功: JWTトークン発行 → メイン画面へ遷移
       └─→ 認証失敗: エラーメッセージ表示

2. タスク管理
   └─→ タスクリスト一覧を表示
       └─→ フィルタ（システム・プロジェクト・進捗状態・デプロイ状態）で絞り込み
       └─→ チケットの追加・編集・削除・登録

3. マスタ管理（管理者のみ）
   └─→ システム・プロジェクト・コード・コードグループ・ユーザーのCRUD操作
```

### 2-3. 業務ルール

- チケットは「システムコード + プロジェクトコード + チケット番号」の複合キーで一意に識別する
- 有効フラグ（`YUKO_FLAG`）で論理削除を管理する（"1"=有効、"0"=無効）
- 進捗状態・デプロイ状態はコードマスタ（M_CODE）で管理し、表示名・カラーをマスタから取得する
- パスワードはbcryptでハッシュ化して保存する（平文保存禁止）
- JWTアクセストークンの有効期限は1時間、リフレッシュトークンは7日間とする

### 3-2. ログイン機能（SC-001）

#### 3-2-1. 機能概要

ユーザーIDとパスワードによる認証を行い、JWTトークンを取得してセッションを確立する。

#### 3-2-2. 画面仕様

| 項目           | 説明                                                       |
|----------------|------------------------------------------------------------|
| 入力項目       | ユーザーID（テキスト）、パスワード（マスク入力）           |
| ログインボタン | クリックまたはEnterキーでログイン処理実行                  |
| エラー表示     | 認証失敗時にエラーメッセージをフォーム内に表示             |
| ローディング   | 通信中はボタンをdisabled化しサインイン中...表示            |
| 成功時         | /Layout へ自動遷移                                         |

#### 3-2-3. 処理仕様

1. ユーザーIDとパスワードを入力してログインボタンを押下
2. `POST /api/userLogin` を呼び出す
3. バックエンドでユーザーIDの存在確認 → 有効フラグ確認 → bcryptパスワード検証
4. 認証成功時:
   - アクセストークン（有効期限1時間）を生成
   - リフレッシュトークン（有効期限7日）を生成し、`T_REFRESH_TOKEN` テーブルへ保存
   - ユーザーID・ユーザー名・ロール・両トークンをレスポンスとして返却
   - フロントエンドは `localStorage` にトークン等を保存
5. 認証失敗時: HTTP 401 とエラーメッセージを返却

#### 3-2-4. バリデーション

- ユーザーIDが未入力の場合はAPIを呼ばない（UI制御）
- 認証失敗時は「ユーザーIDまたはパスワードが正しくありません」と表示する（ブルートフォース対策のため具体的な失敗理由は表示しない）

### 3-3. タスクリスト管理機能（SC-003）

#### 3-3-1. 機能概要

プロジェクトに紐づくチケット（タスク）の一覧表示・追加・編集・削除・登録を行う。

#### 3-3-2. 画面仕様

**テーブルカラム構成**

| # | カラム名       | 入力形式              | 説明                                                               |
|---|---------------|-----------------------|--------------------------------------------------------------------|
| 1 | （選択）       | チェックボックス      | 削除対象の選択                                                     |
| 2 | システム       | セレクトボックス      | システムマスタ（M_SYSTEM）から選択。変更時プロジェクトCDはリセット |
| 3 | プロジェクト   | セレクトボックス      | 選択中のシステムCDに紐づくプロジェクトのみ表示                   |
| 4 | チケット番号   | テキスト入力          | 最大8文字                                                          |
| 5 | リビジョン番号 | テキスト入力（複数行）| 最大50文字                                                         |
| 6 | 進捗状態       | セレクト＋カラーChip  | コードマスタ（GR_CD=001）から選択。カラーが設定された行の背景色を変更 |
| 7 | デプロイ状態   | セレクト＋カラーChip  | コードマスタ（GR_CD=002）から選択。カラーがセルの背景色に反映      |
| 8 | 内容           | テキスト入力（複数行）| 最大1000文字                                                       |
| 9 | 備考           | テキスト入力（複数行）| 最大2000文字                                                       |

**操作ボタン**

| ボタン | 機能                                                       |
|--------|------------------------------------------------------------|
| 登録   | バリデーション確認後、一括保存（確認ダイアログあり）       |
| 追加   | 入力用の空行をテーブル末尾に追加                           |
| 削除   | 選択行を削除（DB保存済みはAPI呼び出し、未保存はローカルのみ）|

**フィルタバー**

| フィルタ項目 | 説明                                                       |
|-------------|-------------------------------------------------------------|
| システム     | システムマスタから選択してフィルタ（全システム分を取得）   |
| プロジェクト | プロジェクトマスタから選択してフィルタ                     |
| 進捗状態     | 進捗状態コードでフィルタ                                   |
| デプロイ状態 | デプロイ状態コードでフィルタ                               |
| 絞込ボタン   | フィルタ条件を確定して一覧を絞り込む（「絞込」ボタン押下時に適用） |

#### 3-3-3. 初期表示データ取得

画面表示時に以下を並列取得する（Promise.all）:

| 取得データ         | API呼び出し                                |
|-------------------|---------------------------------------------|
| タスク一覧         | `POST /api/todoList`                        |
| 進捗状態コード     | `POST /api/getMstCodeCbbList` (grCd=001)    |
| デプロイ状態コード | `POST /api/getMstCodeCbbList` (grCd=002)    |
| プロジェクト一覧   | `POST /api/getMstProjectCbbList`            |
| システム一覧       | `POST /api/getMstSystemCbbList`             |

#### 3-3-4. バリデーション仕様

| フィールド     | 最大文字数 | エラーメッセージ                               |
|----------------|-----------|------------------------------------------------|
| チケット番号   | 8文字     | 「チケット番号は8文字以内で入力してください」  |
| リビジョン番号 | 50文字    | 「リビジョン番号は50文字以内で入力してください」|
| 内容           | 1000文字  | 「内容は1000文字以内で入力してください」       |
| 備考           | 2000文字  | 「備考は2000文字以内で入力してください」       |

### 3-4. マスタ管理機能（SC-004）

マスタ管理画面はタブ形式で5種類のマスタを管理する。

#### 3-4-1. タブ一覧

| タブ# | タブ名           | 管理対象テーブル | 主キー                |
|-------|-----------------|------------------|-----------------------|
| 1     | システム         | M_SYSTEM         | CD                    |
| 2     | プロジェクト     | M_PROJECT        | SYSTEM_CD + CD        |
| 3     | コード           | M_CODE           | GR_CD + CD            |
| 4     | コードグループ   | M_GROUP_CODE     | CD                    |
| 5     | ユーザー         | M_USER           | USER_ID               |

#### 3-4-2. 各マスタ共通仕様

- **一覧表示**: テーブル形式でデータを表示（最大高さスクロール対応）
- **追加**: 「＋追加」ボタンで空行を末尾に追加
- **編集**: テーブルセルを直接編集
- **削除**: チェックボックスで選択後「✕削除」ボタンで削除（確認ダイアログあり）
- **登録**: 「✓登録」ボタンで一括保存（確認ダイアログあり）
- **未保存通知**: 変更がある場合「未保存の変更があります」を表示

#### 3-4-3. システムマスタ（タブ1）

| 項目名     | フィールド    | 型       | 最大長 | 必須 | 説明               |
|------------|-------------|----------|--------|------|--------------------|
| システムCD | cd          | VARCHAR  | 3      | ○   | 主キー             |
| システム名 | systemName  | VARCHAR  | 255    | ○   | システムの名称     |
| 内容       | note        | VARCHAR  | 1000   |      | 説明文             |
| 備考       | biko        | VARCHAR  | 2000   |      | 備考               |
| 有効フラグ | yukoFlag    | CHAR     | 1      | ○   | 1:有効 / 0:無効   |

#### 3-4-4. プロジェクトマスタ（タブ2）

| 項目名         | フィールド   | 型       | 最大長 | 必須 | 説明                     |
|---------------|------------|----------|--------|------|--------------------------|
| システムCD     | systemCd   | VARCHAR  | 3      | ○   | FK → M_SYSTEM.CD         |
| プロジェクトCD | cd         | VARCHAR  | 3      | ○   | プロジェクト識別コード   |
| プロジェクト名 | projectName| VARCHAR  | 100    | ○   | プロジェクトの名称       |
| 内容           | note       | VARCHAR  | 1000   |      | 説明文                   |
| 備考           | biko       | VARCHAR  | 2000   |      | 備考                     |
| 有効フラグ     | yukoFlag   | CHAR     | 1      | ○   | 1:有効 / 0:無効         |

#### 3-4-5. コードマスタ（タブ3）

| 項目名           | フィールド | 型       | 最大長 | 必須 | 説明                       |
|-----------------|----------|----------|--------|------|----------------------------|
| グループコード   | grCd     | VARCHAR  | 3      | ○   | FK → M_GROUP_CODE.CD       |
| コード           | cd       | VARCHAR  | 3      | ○   | コード値                   |
| コード名         | cdName   | VARCHAR  | 3      | ○   | 表示名称                   |
| カラー           | color    | CHAR     | 7      |      | 表示色（#RRGGBB形式）      |
| 内容             | note     | VARCHAR  | 1000   |      | 説明文                     |
| 備考             | biko     | VARCHAR  | 2000   |      | 備考                       |
| 有効フラグ       | yukoFlag | CHAR     | 1      | ○   | 1:有効 / 0:無効           |

#### 3-4-6. コードグループマスタ（タブ4）

| 項目名           | フィールド | 型       | 最大長 | 必須 | 説明                           |
|-----------------|----------|----------|--------|------|--------------------------------|
| グループコード   | cd       | VARCHAR  | 3      | ○   | 主キー                         |
| グループ名       | cdName   | VARCHAR  | 100    | ○   | コードグループの名称           |
| 内容             | note     | VARCHAR  | 1000   |      | 説明文                         |
| 備考             | biko     | VARCHAR  | 2000   |      | 備考                           |
| 有効フラグ       | yukoFlag | CHAR     | 1      | ○   | 1:有効 / 0:無効               |

**コードグループ初期値**

| CD  | 名称       | 用途                    |
|-----|-----------|-------------------------|
| 001 | 進捗状態   | タスクの進捗状態管理    |
| 002 | デプロイ状態 | タスクのデプロイ状態管理 |

#### 3-4-7. ユーザーマスタ（タブ5）

| 項目名           | フィールド    | 型       | 最大長 | 必須 | 説明                                 |
|-----------------|-------------|----------|--------|------|--------------------------------------|
| ユーザーID       | userId      | VARCHAR  | 20     | ○   | ログインID（主キー）                 |
| ユーザー名       | userName    | VARCHAR  | 100    | ○   | 表示名                               |
| パスワード       | plainPassword | -      | -      | △   | 新規登録時必須・更新時は変更のみ入力 |
| ロール           | roleCd      | VARCHAR  | 10     | ○   | ADMIN=管理者 / USER=一般ユーザー     |
| メールアドレス   | email       | VARCHAR  | 255    |      | メールアドレス                       |
| 有効フラグ       | yukoFlag    | CHAR     | 1      | ○   | 1:有効 / 0:無効（アカウントロック）  |

**ユーザーマスタ固有仕様**

- パスワードは bcrypt でハッシュ化して保存（平文は保存しない）
- 既存ユーザーのパスワード欄は「変更時のみ入力」（センチネル値で未変更を識別）
- 一覧表示時にパスワードハッシュはマスク済み（空文字）でレスポンスする
- ロールはドロップダウン＋Chipで視覚的に識別（管理者=赤系、一般=紫系）

**バリデーション**

| フィールド     | 最大長 | その他制約                                    |
|----------------|--------|-----------------------------------------------|
| ユーザーID     | 20文字 |                                               |
| ユーザー名     | 100文字|                                               |
| メールアドレス | 255文字|                                               |
| パスワード     | -      | 新規設定時は最低文字数あり（8文字）           |

### 3-5. AI連携機能

#### 3-5-1. 機能概要

ローカルAIサービス（Ollama、llama3モデル）に対してプロンプトを送信し、AIの応答を取得する。

#### 3-5-2. 提供エンドポイント

| エンドポイント    | 方式             | 説明                                       |
|-----------------|-----------------|---------------------------------------------|
| `POST /mcp/ask` | MCPエンドポイント | プロンプトを送信してAI応答を返す            |
| `GET /api/test5`| テスト用         | Ollama直接呼び出し（クエリパラメータ方式）  |
| `GET /api/test/ai`| テスト用       | McpClientService経由の内部自己呼び出し      |

**リクエスト/レスポンス例**

```json
// リクエスト (POST /mcp/ask)
{ "prompt": "質問内容" }

// レスポンス
{ "answer": "AIの回答テキスト" }
```

#### 3-5-3. 利用するOllama API

- `POST http://localhost:11434/api/generate` （ストリーミング形式）
- `POST http://localhost:11434/v1/chat/completions` （OpenAI互換形式）

## 4. API要件

### 4-1. 共通仕様

| 項目             | 内容                                                         |

| ベースURL         | `http://localhost:8080`                                      |
| 通信プロトコル   | HTTP/REST                                                    |

## 5. 非機能要件

### 5-1. セキュリティ要件

| 項目                 | 要件                                                         |
|----------------------|--------------------------------------------------------------|
| 認証                 | JWT Bearer トークンによる認証（アクセストークン有効期限1時間）|
| パスワード管理       | bcrypt ハッシュ化（平文保存禁止）                           |

| 通信                 | 全 `/api/` パスはJWT認証必須（ログインエンドポイント除く）  |

### 5-2. 可用性・運用要件

| 項目             | 要件                                                         |
|------------------|--------------------------------------------------------------|
| 開発環境         | localhost での動作確認                                       |
| 本番環境         | Docker コンテナ（docker-compose）によるデプロイ              |
| バックアップ     | MySQL のデータバックアップ（別途運用手順を定義）             |

### 5-3. パフォーマンス要件

| 項目             | 目標値                                       |
|------------------|----------------------------------------------|
| 画面初期表示     | 3秒以内（通常利用時）                        |
| タスク登録       | 確認ダイアログ表示 → 完了まで5秒以内         |

### 5-4. 拡張性要件

- インターフェースベースのサービス設計により、実装クラスの差し替えが容易
- コードマスタによるマスタデータ一元管理により、コード値の追加・変更がDBのみで完結
- マスタ画面はタブ構成のため、新マスタ追加時のUI変更が最小化

### 8-1. 未実装・将来実装予定

| ログアウト処理                 | 未実装   | localStorageのクリア処理を要追加            |
| メールアドレス検証            | 未実装   | 書式チェックのバリデーションを要追加        |

### 8-2. 設計上の特記事項

- 全API通信は読み取り操作を含め `POST` メソッドを使用する
- MyBatis Generator により Entity・Mapper を自動生成（手動修正禁止）
- フロントエンドの状態管理は `useState`/`useEffect` のみ（Reduxは使用しない）
- トークン情報はブラウザの `localStorage` に保存する（XSS対策は別途検討が必要）
- JWTの秘密鍵は `application.yml` に定義されており、本番環境では必ず変更すること

### 8-3. 開発環境前提

- Java 17 以上がインストールされていること
- Node.js 18 以上がインストールされていること
- MySQL 8 が `192.168.0.3:3306` で稼働していること
- DB名: `tododb`、接続ユーザー: `yama`
- 本番デプロイ時は Docker / docker-compose が利用可能であること
