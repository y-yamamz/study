================================================================================
1. プロジェクト概要
================================================================================

本アプリケーションは、チケット（タスク）管理を行うTODOリストシステムです。
システムマスタ・プロジェクトマスタに紐づいたチケットを管理し、
進捗状態・デプロイ状態を追跡します。
また、ローカルAI（Ollama）との連携機能も実装されています。


================================================================================
2. システム構成
================================================================================

  [フロントエンド]                [バックエンド]              [データベース]
  React 19 + TypeScript  <--->  Spring Boot 3.5  <--->  MySQL 8
  (localhost:5173)              (localhost:8080)         (192.168.0.17:3306)
                                      |
                                [外部AIサービス]
                                 Ollama (llama3)
                                (localhost:11434)

通信方式: REST API (JSON)
CORS設定: フロントエンドのオリジン (http://localhost:5173) を許可

3-3. APIエンドポイント一覧
--------------------------------------------------------------------------------

ベースURL: http://localhost:8080

[ユーザー・認証]
  POST /api/userLogin
    説明    : ユーザーログイン
    リクエスト: UserForm { id: string, name: string, password: string }
    レスポンス: "OK"

[メニュー]
  POST /api/menuList
    説明    : メニューリスト取得
    リクエスト: MenuListDao { id, name, url }
    レスポンス: List<MenuListDao>

[TODOリスト]
  POST /api/todoList
    説明    : TODOリスト全件取得（システム名・プロジェクト名・コード名を結合）
    リクエスト: なし（空ボディ）
    レスポンス: List<TodoListDataDto>
      - systemCd, projectCd, ticketNo (複合主キー)
      - systemName, projectName (マスタから結合)
      - statusCd, statusNm (進捗状態コード・名称)
      - deployCd, deployNm (デプロイ状態コード・名称)
      - revisionNo, note, biko, yukoFlag
      - insDate, insUserId, updDate, updUserId

  POST /api/todoRegister
    説明    : TODOリスト登録・更新（一括保存）
    リクエスト: List<TodoListDataDto>
    レスポンス: { status: "OK", message: "登録完了" }

[マスタ管理]
  POST /api/getMstSystemCbbList
    説明    : システムマスタ コンボボックス用リスト取得
    リクエスト: なし
    レスポンス: List<MSystemDto> { cd, systemName }

  POST /api/getMstProjectCbbList
    説明    : プロジェクトマスタ コンボボックス用リスト取得（システムコードでフィルタ）
    リクエスト: MProjectDto { systemCd, cd, projectName }
    レスポンス: List<MProjectDto> { systemCd, cd, projectName }

  POST /api/getMstCodeCbbList
    説明    : コードマスタ コンボボックス用リスト取得（グループコードでフィルタ）
    リクエスト: MCodeDto { grCd, cd, cdName }
    レスポンス: List<MCodeDto> { grCd, cd, cdName }
      - grCd="001" : 進捗状態コード
      - grCd="002" : デプロイ状態コード

[AI連携]
  POST /mcp/ask
    説明    : Ollama AIへの問い合わせ（MCPエンドポイント）
    リクエスト: { prompt: string }
    レスポンス: { answer: string }

  GET /api/test5?prompt={prompt}
    説明    : Ollama AIへの直接問い合わせ（テスト用）
    リクエスト: クエリパラメータ prompt
    レスポンス: AIの生成テキスト

  GET /api/test/ai
    説明    : McpClientServiceのテスト
    レスポンス: AIの応答テキスト

[テスト用エンドポイント]
  POST /api/test   → "OK" を返す
  POST /api/test2  → "OK" を返す
  POST /api/test3  → メニュー件数を含む文字列
  POST /api/test4  → List<MenuListDao>

  カラム名    | 型        | 説明

  systemName  | VARCHAR   | システム名
  note        | VARCHAR   | 内容

  updDate     | TIMESTAMP | 更新日時

【M_PROJECT テーブル】（複合主キー: systemCd + cd）
  カラム名     | 型        | 説明
  -------------|-----------|----------------------------------
  systemCd     | VARCHAR   | システムコード (FK → M_SYSTEM)
  cd           | VARCHAR   | プロジェクトコード

テーブルリレーション図:

  M_CODE(grCd=001) ──< TODO_LIST.statusCd
  M_CODE(grCd=002) ──< TODO_LIST.deployCd


--------------------------------------------------------------------------------
3-5. サービス層
--------------------------------------------------------------------------------

[TodoListService / TodoListServiceImpl]
  getListData()
    → TODO_LIST を M_SYSTEM・M_PROJECT・M_CODE(2回) と結合して全件取得
    → 結合SQL: INNER JOIN M_SYSTEM, INNER JOIN M_PROJECT,
               LEFT JOIN M_CODE(statusCd), LEFT JOIN M_CODE(deployCd)
  save(List<TodoListDataDto>)
    → TODOリストの一括登録・更新
    → 登録日時(insDate)・更新日時(updDate)・ユーザーIDを自動セット

[MMenuService / MMenuServiceImle]
  getListData()
    → M_MENU からメニューリスト全件取得

[MstSystemService / MstSystemServiceImpl]
  getCbbList()
    → M_SYSTEM からコンボボックス用データを取得

[MstProjectService / MstProjectServiceImpl]
  getCbbList(MProjectDto)
    → systemCd でフィルタして M_PROJECT からリスト取得

[MstCodeService / MstCodeServiceImpl]
  getCbbList(MCodeDto)
    → grCd でフィルタして M_CODE からリスト取得

[OllamaService / OllamaServiceImpl]
  generate(String model, String prompt)
    → POST http://localhost:11434/api/generate （ストリーミング形式）
    → RestTemplate を使用してローカルOllama APIを呼び出す
  chat(String prompt)
    → POST http://localhost:11434/v1/chat/completions
    → llama3 モデルを使用（OpenAI互換API形式）

[McpClientService / McpClientServiceImpl]
  askServer(String prompt)
    → 内部エンドポイント /mcp/ask を HTTP呼び出し（自己呼び出し）

サービス設計パターン:
  - インターフェースと実装クラスを分離（Interface-based）
  - Lombok @RequiredArgsConstructor でコンストラクタインジェクション
  - @Service アノテーションでSpring Bean登録

  - データ形式  : JSON
  - Content-Type: application/json

フロントエンドでのAPI呼び出しパターン:
  const response = await fetch(`${SERVICE_URL.BASE_URL}api/xxxx`, {
    method: "POST",

    body: JSON.stringify(requestData),
  });
  const data = await response.json();

  - try/catch ブロックで例外捕捉

  - 登録前に window.confirm() で確認ダイアログ表示

6. アーキテクチャ概要
================================================================================

バックエンドアーキテクチャ（レイヤー構成）:
  ┌─────────────────────────────────┐
  │  Controller Layer (REST API)     │  リクエスト受付・バリデーション
  ├─────────────────────────────────┤
  │  Service Layer (Business Logic)  │  ビジネスロジック・トランザクション
  ├─────────────────────────────────┤
  │  Mapper Layer (MyBatis)          │  SQL実行・DBアクセス
  ├─────────────────────────────────┤
  │  Database (MySQL)                │  データ永続化
  └─────────────────────────────────┘

フロントエンドアーキテクチャ（コンポーネント構成）:
  ┌─────────────────────────────────┐
  │  App.tsx (Router)                │  ルーティング定義
  ├─────────────────────────────────┤
  │  Layout.tsx                      │  レイアウト（サイドバー + メイン）
  ├────────────┬────────────────────┤
  │ Header.tsx │  Page Components    │  各機能ページ
  │            │  (TaskListPage etc) │
  └────────────┴────────────────────┘

設計上の特徴:
  - MyBatisGeneratorによるエンティティ・マッパーの自動生成
  - エンティティクラスに Key クラス（複合主キー）・Example クラス（クエリビルダー）を含む継承構造
  - DTOパターン: エンティティを継承しフロントエンド表示用フィールドを追加
  - インターフェースベースのサービス設計（差し替え容易）
  - マスタデータはコードマスタ（M_CODE）に集約し、グループコードで種別分け
  - 論理削除: yukoFlag カラムで有効/無効を管理
  - 監査カラム: insDate, insUserId, updDate, updUserId を全テーブルに実装
  - シンプルな状態管理: React useState/useEffect のみ使用（Reduxなし）
  - AIとの連携: Ollama（ローカルLLM）を2種類のAPIで呼び出し

開発中・未実装箇所:
  - MasterPage.tsx  : マスタデータ管理画面（未実装）

  - 認証・セッション管理: ログイン後のセッション保持機能なし
